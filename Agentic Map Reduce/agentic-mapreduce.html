Architecture Deep Dive · July 2026

Agentic MapReduce

Cognition's new architecture behind Devin Security Swarm — how to make an agent reason about every file in a 50k-file repo without paying for an agent to read every file. A two-decade-old distributed-systems idea, with one clever inversion.

First, feel the problem

The census analogy

Imagine you must count every house in Sri Lanka with a leaking roof. You could send one brilliant inspector to wander the island — but she'd spend 90% of her time walking between houses, her notebook would overflow, and when she says "I think I've seen them all," you can't verify it. Or: a planner first pulls the land registry (a deterministic list of every house), splits it into districts, sends one inspector per district in parallel, and a supervisor merges their reports. Coverage isn't a feeling — it's guaranteed by the registry.

Coding agents have exactly this problem. Point a single grep-and-read agent at a huge repo and ask "find every SQL injection" — three things break:

FAILURE 01

Budget burns on finding, not doing

The agent greps, opens wrong files, backtracks. On big repos, selection dominates analysis — one study found reading/searching ate 56% of tool-use turns.

FAILURE 02

Context becomes the bottleneck

One long-running agent drags discoveries from file 1 while reasoning about file 4,000. Unrelated evidence competes for attention.

FAILURE 03

No coverage boundary

A search agent stops when it feels done — not when a finite work queue is exhausted. "I've looked everywhere" is unfalsifiable.

The classic idea, one inversion

Hadoop-era MapReduce: shard a giant input, run the same handwritten map function over every shard in parallel, reduce results into one answer. It made cluster-scale data processing affordable.

Agentic MapReduce keeps the skeleton but flips who writes the map. In classic MapReduce, a human writes the instructions and dumb machines execute over everything. Here, an agent writes a deterministic relevance test (a "selector") for this specific repo and task — then cheap, model-free code executes it over everything. Reasoning is spent once, when the selector is authored.

StageWhat happensAgentic?
PlanAn agent studies the repo and authors selectors — relevance tests concrete enough to run without a model (Tree-sitter queries, compiler symbol queries, import-graph traversals, lexical patterns)YES ✦

ReduceAn agent consumes workers' compressed conclusions (not transcripts), dedupes, reconciles, and finds cross-shard relationshipsYES ✦
Why coverage is now provable

The deterministic pass produces a finite work queue. Every shard is assigned to a worker; the scan is complete only when the queue is exhausted. The trade-off shifts to selector recall — a file matching no selector never reaches a worker. But selectors are an inspectable, version-controlled artifact: you can read them, test them against known examples, and tune them. A search agent's "I've looked everywhere" cannot be audited; a selector can.

How Devin uses it: Security Swarm

Launched July 1, 2026, Devin Security Swarm is the first product built on this pattern — and security scanning is the shape it was designed for: a vulnerability report is only trustworthy if the whole codebase was considered, real bugs are sparse, and the worst exploits chain across files that no single worker sees together. Swarm adds a fifth stage: Verify.

Why the Reduce stage is the quiet hero

Each Map worker only sees its shard. The Reducer holds the global view no worker had — so it can compose an unauthenticated ID leak from shard 2 with an ID-gated RCE from shard 7 into a single P0 attack chain. For migrations, the same stage groups affected call sites from many shards under the API change that caused them and emits one migration plan.

Why it's cheap — and stays cheap

Cost tracks the amount of relevant code, not repo size: tokens are spent only on candidates that survive the selectors, one bounded shard at a time, and the Reducer reasons over compressed conclusions rather than transcripts. On re-runs, the pipeline only touches files changed since the last scanned commit — you pay for the diff, not a full pass. Selectors are persisted and reused.

A second walkthrough: retiring a deprecated API

Security scanning is the flagship, but nothing about Plan → Shard → Map → Reduce is specific to vulnerabilities — it's a general answer to "is this file relevant, and what should happen to it?" Point the same four stages at "does this file call a deprecated API" instead, and you get a migration engine. Only the selector, the Map verdict, and what Reduce goes looking for change. (Cognition hasn't shipped a packaged product for this — but the pipeline underneath doesn't know the difference.)

The old-building analogy

Picture a 1930s office building getting rewired for modern code: instead of one electrician tracing every wire from basement to roof, you split the job floor by floor and send a crew to each, in parallel. That's safe almost everywhere — a floor's outlets and switches are self-contained. The danger lives at the risers, where floor 4's wiring ties into floor 3's breaker panel. A floor-by-floor crew can rewire their own floor flawlessly and still leave a live breaker feeding a rewired outlet one floor up — no single crew would ever see that. It takes the inspector holding the whole building's blueprint. In Agentic MapReduce that's the Reduce stage; in code, the risers are shard boundaries.

Here's a representative walkthrough — illustrative, not a published case study, but a shape every long-lived codebase eventually hits. Call the repo Helios: 38,000 files, six services, and years of calling datetime.utcnow() to stamp events in UTC. Python deprecated it in 3.12 — it returns a naive datetime, no timezone attached, which looks correct right up until it collides with an aware one. A repo-wide search turns up 1,900 call sites. Rewriting one is a one-line diff; rewriting 1,900 without quietly breaking the ones that depend on the old naive behaviour is the actual job.

StageWhat happens for this migrationAgentic?
PlanWrites an AST selector for utcnow()/utcfromtimestamp() (Python's own ast module, no model) and a rulebook of transformations a Map worker may apply on its ownYES ✦
ShardSelector walks the AST of every file once; 1,900 matches bucketed into 40 batches of roughly 48, grouped by serviceno
Map40 child agents in parallel apply the rulebook to their batch, reading the real function around each call site, and emit one verdict per matchYES ✦
ReduceCollects every verdict, walks the import graph one hop out from every changed function to catch what the selector structurally can't see, and emits one migration planYES ✦

Why Reduce has to hold the blueprint

A selector built from utcnow()'s spelling only finds call sites of that exact name — it can't find every place downstream that later touches the value, since that means tracing data flow through the whole program, exactly the expense this pattern avoids. So two Map workers, in two different batches, each do their job correctly and still leave a live bug between them:

# shard 12 · matched the selector
def issue_invoice(order):
    return {
        "issued_at": datetime.now(timezone.utc),
    }
# shard 27 · never matched, no utcnow() here
def days_since_issued(issued_at):
    return (datetime.now() - issued_at).days

Worker 12 is right: issue_invoice now returns a timezone-aware datetime, exactly as the rulebook says. Worker 27's shard was never even in the queue — aging.py doesn't call utcnow() anywhere, so no selector matched it. But datetime.now() with no argument returns naive local time, nothing to do with UTC, and the moment that naive value meets the newly-aware issued_at in a subtraction, Python refuses outright.

Neither worker did anything wrong — neither could see the other's file, which is the entire point of sharding. Reduce is the one component that spans shards by design, but re-reading the whole repo would put the pattern right back where it started. What it can do instead: one narrow second deterministic pass. For every changed function, walk the already-built import graph one hop out to its callers, then run a cheap lexical check — does this file do any datetime arithmetic at all — before flagging it for a closer look. On Helios that surfaces roughly 300 second-pass candidates, aging.py among them. It isn't proof that every boundary bug is caught; only whole-program data-flow analysis could promise that, and that's precisely the bill this pattern refuses to pay. Same trade as the file-level pass, one hop further out: bounded and honest about its edges beats exhaustive and unaffordable.

Same bet as Security Swarm, aimed at a different question: spend agent judgment once — on the selector, the rulebook, and the edges it's known not to reach — then let deterministic code carry that judgment across every file, for free. Verify closes it out the same cheap way: run the patch's tests plus a strict type-checker (which catches exactly this naive/aware class of bug) in a sandbox, and only what passes rides along in the PR.

The mental model to keep

CLASSIC MAPREDUCE

Human writes the map

Handwritten function, dumb executors, full input processed. Scales computation.

NAIVE MULTI-AGENT

Agents everywhere

Agents search, agents coordinate, agents talk to agents. Expensive, unbounded, unauditable coverage.

AGENTIC MAPREDUCE

Agent writes the map

Reasoning spent once on selectors; deterministic pass guarantees coverage; parallel agents only where judgment is needed.

This also squares with Cognition's earlier "Don't Build Multi-Agents" stance: the pattern works because workers never coordinate with each other. They share no context, make no conflicting decisions — the deterministic shard boundary does the coordination, and a single Reducer does the synthesis.

It fits any task where a verdict is only trustworthy if the whole corpus was in view: security scans, breaking-change detection, code-quality enforcement, large-scale migrations — and beyond code, arguably any bounded-corpus audit.

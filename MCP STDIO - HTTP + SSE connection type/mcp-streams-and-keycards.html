Question 1 · stdio

The client launches the server. So the pipes belong to the server.

In a stdio connection the client spawns your MCP server as a subprocess. That single fact decides everything: stdin, stdout and stderr are the server's three streams, so name them from the server's point of view and the directions stop being confusing.

Three streams flowing. stdin in, stdout out, stderr to the side.
Live: requests travel in on stdin, results out on stdout, logs drop into stderr.

stdin — the server reads here, and the client writes to it. Everything heading into the server flows down this pipe: JSON-RPC requests and notifications from the client.

stdout — the server writes here, and the client reads from it. Everything coming back rides this pipe: responses and server-initiated notifications.

stderr — the server's private side channel for logs and diagnostics. It is free-form text, and the client is allowed to capture it, forward it, or throw it away.

the one rule that bites everyone stdout is sacred — it may only carry valid MCP messages. Messages are newline-delimited JSON-RPC, one per line, with no embedded newlines. A stray print(), a debug line, or an uncaught traceback landing on stdout is like coughing into an intercom that only accepts a strict format: the client can't parse it, and the connection drops. That is why every log goes to stderr. (Press the button above to watch it happen.)

The intuition: stdin/stdout are an official intercom that only speaks one rigid format, while stderr is the scratchpad the server mutters onto. No network, no ports, no auth — when the client exits, the OS reaps the subprocess. That isolation is exactly why stdio is the default for local tools.

Question 2 · HTTP auth

API key, Bearer, OAuth — three layers, not three rivals.

a correction worth making first Bearer tokens, OAuth and API keys aren't features of SSE specifically. They're HTTP-layer auth, so they apply to any HTTP transport — the legacy HTTP+SSE and the current Streamable HTTP alike. stdio skips all of it: a local subprocess just inherits credentials from the environment it was launched in.

The trap is reading these as three competing options. They live at different layers. The cleanest way to feel the difference is a hotel keycard.

01
what you send

API key

The master key you cut yourself and hand to the cleaner. Works forever — until you re-key the locks.

A static secret string, usually in a custom header. No expiry by default, no notion of which person is calling unless you map keys to identities. Simplest to ship; riskiest if it leaks.

02
how it travels

Bearer token

The hotel keycard. Whoever bears it gets in — the door doesn't re-check your face. Expires at checkout.

The standardised envelope: Authorization: Bearer <token>. Often a JWT that's self-describing — who you are, which scopes, when it expires. The spec says it goes in the header, never a URL query string.

how it's issued

OAuth 2.1

The front desk. You show your passport once and consent; the desk mints the keycard. The cleaner never sees your passport.

Not a credential — a handshake that produces a bearer token. An authorization server issues the token after the user logs in and approves scopes. This is what MCP mandates for public remote servers.

how this lands in MCP The spec mandates OAuth 2.1 with PKCE for public remote HTTP servers; a static bearer token or API key is fine for internal and team tools; stdio needs none. Two guardrails the spec is strict about: the token never rides in a URL query string, and the MCP server must validate that a token was issued for itself — it must not blindly forward your token to an upstream API.
The 30-second map

Pick by where the server lives.

stdio

local · same machine
  • Client spawns server as a subprocess
  • stdin = in, stdout = out, stderr = logs
  • stdout carries valid JSON-RPC only
  • No network, no ports, no auth
  • Credentials come from the environment

HTTP (Streamable HTTP)

remote · across the network
  • One endpoint, POST + optional SSE stream
  • Auth lives at the HTTP layer
  • Public servers: OAuth 2.1 + PKCE
  • Internal tools: bearer token / API key
  • Bearer header always; never a query string
